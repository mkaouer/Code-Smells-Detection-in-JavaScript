13     package edu.cmu.sphinx.util.props;
14     
15     /**
16      * Defines the interface that must be implemented by any configurable component in Sphinx-4.  The life cycle of a
17      * component is as follows:
18      * <p/>
19      * <ul><li> <b>Class Parsing</b> The class file is parsed in order to determine all its configurable properties.  These
20      * are defined using <code>public static final String</code> fields which are annotated with one of the following
21      * annotations: <ul> <li>S4Integer <li>S4Double <li>S4Boolean <li>S4Component <li>S4ComponentList </ul> Further
22      * informations about property-specific fields can be found in the javadoc of the property-annotation-definitions. Only
23      * names of annotated properties will be allowed by the configuration system later on.
24      * <p/>
25      * <li> <b>Construction</b> - The (empty) component constructor is called in order to instantiate the component.
26      * Typically the constructor does little, if any work, since the component has not been configured yet.
27      * <p/>
28      * <li> <b> Configuration</b> - Shortly after instantiation, the component's <code>newProperties</code> method is
29      * called. This method is called with a <code>PropertySheet</code> containing the properties (usually taken from an
30      * external configuration file). The component should extract the properties from the property sheet. If some properties
31      * defined for a component does not fullfill the property definition given by the annotation (type, range, etc.) a
32      * <code>PropertyException</code> is thrown. Typically, once a component gets its configuration data via the
33      * <code>newData</code> method, the component will initialize itself.
34      * <p/>
35      * Note: In most cases <code>newProperties</code> is called only once as a result of system configuration during
36      * startup. But nevertheless it is possible (and sometimes necessary) to reconfigure a component while it's running.
37      * Therefore, a well behaved component should react properly to multiple <code>newProperties</code> calls. </ul>
38      * <p/>
39      * <p><b>Connecting to other components</b> <p> Components often need to interact with other components in the system.
40      * One of the design goals of Sphinx-4 is that it allows for very flexible hook up of components in the system.
41      * Therefore, it is *not* considered good S4 style to hardcode which subcomponents a particular subcomponent is
42      * interacting with.  Instead, the component should use the configuration manager to provide the hook up to another
43      * component.
44      * <p/>
45      * For example, if a component needs to interact with a Linguist. Instead of explicitly setting which linguist is to be
46      * used via a constructor or via a <code>setLinguist</code> call, the component should instead define a configuration
47      * property for the linguist.  This would be done like so:
48      * <p/>
49      * <code> <pre>
50      *     \@S4Component(type=Linguist.class)
51      *     public static String PROP_LINGUIST = "linguist";
52      * </pre> </code>
53      * <p> The linguist is made available in the <code>newProperties</code> method, like so: <p>
54      * <code> <pre>
55      *     public void newProperties(PropertySheet propertySheet) {
56      *      linguist = (Linguist) propertySheet.getComponent(PROP_LINGUIST);
57      *     }
58      * </pre> </code>
59      * <p/>
60      * This <code>getComponent</code> call will find the proper linguist based upon the configuration data.  Thus, if the
61      * configuration for this component had the 'linguist' defined to be 'dynamicLexTreeLinguist', then the configuration
62      * manager will look up and return a linguist with that name, creating and configuring it as necessary.  Of course, the
63      * dynamicLexTreeLinguist itself may have a number of sub-components that will be created and configured as a result. If
64      * the component doesn't exist (but was defined to mandatory) and no configuration information is found in the config
65      * file for it, or if it is of the wrong type, a <code>PropertyException</code> will be thrown.
66      */
67     public interface Configurable {
68     
69     
70         /**
71          * This method is called when this configurable component needs to be reconfigured.
72          *
73          * @param ps a property sheet holding the new data
74          * @throws PropertyException if there is a problem with the properties.
75          */
76         public void newProperties(PropertySheet ps) throws PropertyException;
77     }
