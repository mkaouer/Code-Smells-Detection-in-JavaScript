14     
15     import edu.cmu.sphinx.util.props.Configurable;
16     import edu.cmu.sphinx.util.props.S4Double;
17     
18     import java.io.IOException;

20     /**
21      * The linguist is responsible for representing and managing the search space for the decoder.  The role of the linguist
22      * is to provide, upon request, the search graph that is to be used by the decoder.  The linguist is a generic interface
23      * that provides language model services.
24      * <p/>
25      * The main role of any linguist is to represent the search space for the decoder. The search space can be retrieved by
26      * a SearchManager via a call to <code> getSearchGraph</code>. This method returns a SearchGraph. The initial state in
27      * the search graph can be retrieved via a call to <code>getInitialState</code> Successor states can be retrieved via
28      * calls to <code>SearchState.getSuccessors().</code>. There are a number of search state subinterfaces that are used to
29      * indicate different types of states in the search space:
30      * <p/>
31      * <ul> <li><b>WordSearchState </b>- represents a word in the search space. <li><b>UnitSearchState </b>- represents a
32      * unit in the search space <li><b>HMMSearchState </b> represents an HMM state in the search space </ul>
33      * <p/>
34      * A linguist has a great deal of latitude about the order in which it returns states. For instance a 'flat' linguist
35      * may return a WordState at the beginning of a word, while a 'tree' linguist may return WordStates at the ending of a
36      * word. Likewise, a linguist may omit certain state types completely (such as a unit state). Some Search Managers may
37      * want to know a priori the order in which different state types will be generated by the linguist. The method
38      * <code>SearchGraph.getNumStateOrder()</code> can be used to retrieve the number of state types that will be returned
39      * by the linguist. The method <code>SearchState.getOrder()</code> returns the ranking for a particular state.
40      * <p/>
41      * <p/>
42      * Depending on the vocabulary size and topology, the search space represented by the linguist may include a very large
43      * number of states. Some linguists will generate the search states dynamically, that is, the object representing a
44      * particular state in the search space is not created until it is needed by the SearchManager. SearchManagers often
45      * need to be able to determine if a particular state has been entered before by comparing states. Because SearchStates
46      * may be generated dynamically, the <code>SearchState.equals()</code> call (as opposed to the reference equals '=='
47      * method) should be used to determine if states are equal. The states returned by the linguist will generally provide
48      * very efficient implementations of <code>equals</code> and <code>hashCode</code>. This will allow a SearchManager to
49      * maintain collections of states in HashMaps efficiently.
50      * <p/>
51      * <p/>
52      * <p/>
53      * The lifecycle of a linguist is as follows: <ul>
54      * <p/>
55      * <li> The linguist is created by the configuration manager
56      * <p/>
57      * <li> The linguist is given an opportunity to register its properties via a call to its <code>register</code> method.
58      * <p/>
59      * <li>  The linguist is given a new set of properties via the <code>newProperties</code> call.  A well written linguist
60      * should be prepared to respond to <code>newProperties</code> call at any time.
61      * <p/>
62      * <li> The <code>allocate</code> method is called. During this call the linguist generally allocates resources such as
63      * acoustic and language models. This can often take a significant amount of time. A well-written linguist will be able
64      * to deal with multiple calls to <code>allocate</code>. This can happen if a linguist is shared by multiple search
65      * managers.
66      * <p/>
67      * <li> The <code>getSearchGraph</code> method is called by the search to retrieve the search graph that is used to
68      * guide the decoding/search.  This method is typically called at the beginning of each recognition. The linguist should
69      * endeavor to return the search graph as quickly as possible to reduce any recognition latency.  Some linguists will
70      * pre-generate the search graph in the <code>allocate</code> method, and only need to return a reference to the search
71      * graph, while other linguists may dynamically generate the search graph on each call.  Also note that some linguists
72      * may change the search graph between calls so a search manager should always get a new search graph before the start
73      * of each recognition.
74      * <p/>
75      * <li> The <code>startRecognition</code> method is called just before recognition starts. This gives the linguist the
76      * opportunity to prepare for the recognition task.  Some linguists may keep caches of search states that need to be
77      * primed or flushed. Note however that if a linguist depends on <code>startRecognition</code> or
78      * <code>stopRecognition</code> it is likely to not be a reentrant linguist which could limit its usefulness in some
79      * multi-threaded environments.
80      * <p/>
81      * <li> The <code>stopRecognition</code> method is called just after recognition completes. This gives the linguist the
82      * opportunity to cleanup after the recognition task.  Some linguists may keep caches of search states that need to be
83      * primed or flushed. Note however that if a linguist depends on <code>startRecognition</code> or
84      * <code>stopRecognition</code> it is likely to not be a reentrant linguist which could limit its usefulness in some
85      * multi-threaded environments.
86      * <p/>
87      * </ul>
88      */
89     public interface Linguist extends Configurable {
90     
91         /** Word insertion probability property */
92         @S4Double(defaultValue = 1.0)
93         public final static String PROP_WORD_INSERTION_PROBABILITY = "wordInsertionProbability";
94     
95         /** Unit insertion probability property */
96         @S4Double(defaultValue = 1.0)
97         public final static String PROP_UNIT_INSERTION_PROBABILITY = "unitInsertionProbability";
98     
99         /** Silence insertion probability property */
100         @S4Double(defaultValue = 1.0)
101         public final static String PROP_SILENCE_INSERTION_PROBABILITY = "silenceInsertionProbability";
102     
103         /** Filler insertion probability property */
104         @S4Double(defaultValue = 1.0)
105         public final static String PROP_FILLER_INSERTION_PROBABILITY = "fillerInsertionProbability";
106     
107         /** Sphinx property that defines the language weight for the search */
108         @S4Double(defaultValue = 1.0)
109         public final static String PROP_LANGUAGE_WEIGHT = "languageWeight";
110     
111     
112         /**
113          * Retrieves search graph.  The search graph represents the search space to be used to guide the search.
114          * <p/>
115          * <p/>
116          * Implementor's note: This method is typically called at the beginning of each recognition and therefore should be
117          *
118          * @return the search graph
119          */
120         public SearchGraph getSearchGraph();
121     
122     
123         /**
124          * Called before a recognition. This method gives a linguist the opportunity to prepare itself before a recognition
125          * begins.
126          * <p/>
127          * <p/>
128          * Implementor's Note - Some linguists (or underlying lanaguge or acoustic models) may keep caches or pools that
129          * need to be initialzed before a recognition. A linguist may implement this method to perform such initialization.
130          * Note however, that an ideal linguist will, once allocated, be state-less. This will allow the linguist to be
131          * shared by multiple simulataneous searches. Reliance on a 'startRecognition' may prevent a linguist from being
132          * used in a multi-threaded search.
133          */
134         public void startRecognition();
135     
136     
137         /**
138          * Called after a recognition. This method gives a linguist the opportunity to clean up after a recognition has been
139          * completed.
140          * <p/>
141          * <p/>
142          * Implementor's Note - Some linguists (or underlying lanaguge or acoustic models) may keep caches or pools that
143          * need to be flushed after a recognition. A linguist may implement this method to perform such flushing. Note
144          * however, that an ideal linguist will once allocated, be state-less. This will allow the linguist to be shared by
145          * multiple simulataneous searches. Reliance on a 'stopRecognition' may prevent a linguist from being used in a
146          * multi-threaded search.
147          */
148         public void stopRecognition();
149     
150     
151         /**
152          * Allocates the linguist. Resources allocated by the linguist are allocated here. This method may take many seconds
153          * to complete depending upon the linguist.
154          * <p/>
155          * <p/>
156          * Implementor's Note - A well written linguist will allow allocate to be called multiple times without harm. This
157          * will allow a linguist to be shared by multiple search managers.
158          *
159          * @throws IOException if an IO error occurs
160          */
161         public void allocate() throws IOException;
162     
163     
164         /**
165          * Deallocates the linguist. Any resources allocated by this linguist are released.
166          * <p/>
167          * <p/>
168          * Implementor's Note - if the linguist is being shared by multiple searches, the deallocate should only actually
169          * deallocate things when the last call to deallocate is made. Two approaches for dealing with this:
170          * <p/>
171          * (1) Keep an allocation counter that is incremented during allocate and decremented during deallocate. Only when
172          * the counter reaches zero should the actually deallocation be performed.
173          * <p/>
174          * (2) Do nothing in dellocate - just the the GC take care of things
175          */
176         public void deallocate();
177     }
