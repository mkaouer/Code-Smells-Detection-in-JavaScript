10       <H1>CppUnit Cookbook</H1>
11       <P>
12         Here is a short cookbook to help you get started.
13         Note that all CppUnit types are defined in the CppUnit
14         namespace.  For brevity, the prefix <b>CppUnit::</b>
15         is omitted in the following examples.
16       </P>
17       <H2>Simple Test Case</H2>
18       <P>
19         You want to know whether your code is working. 
20         How do you do it? 
21         There are many ways. Stepping through a debugger or 
22         littering your code with stream output calls are two of 
23         the simpler ways, but they both have drawbacks. 
24         Stepping through your code is a good idea, but it 
25         is not automatic. You have to do it every time you
26         make changes. Streaming out text is also fine, 
27         but it makes code ugly and it generates far more 
28         information than you need most of the time.
29       </P>
30       <P>
31         Tests in CppUnit can be run automatically. 
32         They are easy to set up and once you have 
33         written them, they are always there to help 
34         you keep confidence in the quality of your code.
35       </P>
36       <P>
37         To make a simple test, here is what you do:
38       </P>
39       <P>
40         Subclass the TestCase class. 
41         Override the method <code>runTest ()</code>. When you 
42         want to check a value, call <code>CPPUNIT_ASSERT (bool)</code> 
43         and pass in an expression that is true if the 
44         test succeeds. 
45       </P>
46       <P>
47         For example, to test the equality comparison 
48         for a Complex number class, write:
49       </P>
50       <TT><PRE>&#9;class ComplexNumberTest : public TestCase {&nbsp;
51     &#9;public: 
52                         ComplexNumberTest (string name) : TestCase (name) {}
53             void        runTest () {
54                             CPPUNIT_ASSERT (Complex (10, 1) == Complex (10, 1));
55                             CPPUNIT_ASSERT (!(Complex (1, 1) == Complex (2, 2)));
56                         }
57             };</PRE>
58         </TT>
59         <P>
60           That was a very simple test. Ordinarily, 
61           you'll have many little test cases that you'll 
62           want to run on the same set of objects. To do 
63           this, use a fixture.
64         </P>
65         <P>&nbsp;</P>
66         <H2>Fixture</H2>
67         <P>
68           A fixture is a known set of objects that 
69           serves as a base for a set of test cases. 
70           Fixtures come in very handy when you are 
71           testing as you develop. 
72           Let's try out this style of development and 
73           learn about fixtures along the away. Suppose 
74           that we are really developing a complex 
75           number class. Let's start by defining a 
76           empty class named Complex.
77         </P>
78         <TT><PRE>&#9;class Complex {};&nbsp;</PRE>
79         </TT>
80         <P>Now create an instance of ComplexNumberTest 
81           above, compile the code and see what happens. 
82           The first thing we notice is a few compiler 
83           errors. The test uses operator==, but it is 
84           not defined. Let's fix that.
85         </P>
86         <TT><PRE>&#9;bool operator== (const Complex&amp; a, const Complex&amp; b) { return true; }</PRE>
87         </TT>
88         <P>
89           Now compile the test, and run it. This time it 
90           compiles but the test fails. 
91           We need a bit more to get an operator== working 
92           correctly, so we revisit the code.
93         </P>
94         <TT><PRE>&#9;class Complex {&nbsp;
95             friend bool operator== (const Complex&amp; a, const Complex&amp; b);
96             double      real, imaginary;
97             public:
98                         Complex ( double r, double i = 0 ) : real(r),imaginary(i) {}
99             };
100     
101             bool operator== (const Complex&amp; a, const Complex&amp; b)
102             { return eq(a.real,b.real) &amp;&amp; eq(a.imaginary,b.imaginary); }</PRE>
103         </TT>
104         <P>
105           If we compile now and run our test it will pass. 
106         </P>
107         <P>
108           Now we are ready to add new operations and 
109           new tests. At this point a fixture would be 
110           handy. We would probably be better off when 
111           doing our tests if we decided to instantiate 
112           three or four complex numbers and reuse them 
113           across our tests. 
114         </P>
115         <P>
116           Here is how we do it:
117         </P>
118         <OL>
119           <LI>Add member variables for each part of the fixture </LI>
120           <LI>Override "setUp ()" to initialize the variables </LI>
121           <LI>Override "tearDown ()" to release any permanent resources you allocated in "setUp ()"</LI>
122         </OL>
123         <TT><PRE>&#9;class ComplexNumberTest : public TestCase  {
124     &#9;private:
125             Complex &#9;*m_10_1, *m_1_1; *m_11_2;
126     &#9;protected:
127     &#9;void&#9;&#9;setUp ()  {
128     &#9;&#9;&#9;    m_10_1 = new Complex (10, 1);
129     &#9;&#9;&#9;    m_1_1  = new Complex (1, 1);
130     &#9;&#9;&#9;    m_11_2  = new Complex (11, 2);  
131                             }
132     &#9;void&#9;&#9;tearDown ()  {
133     &#9;&#9;&#9;    delete m_10_1, delete m_1_1, delete m_11_2;
134     &#9;&#9;&#9;}
135     &#9;};</PRE>
136         </TT>
137         <P>
138           Once we have this fixture, we can add the complex 
139           addition test case any any others that we need 
140           over the course of our development.
141         </P>
142         <P>&nbsp;</P>
143         <H2>Test Case</H2>
144         <P>
145           How do you write and invoke individual tests using a fixture? 
146         </P>
147         <P>
148           There are two steps to this process:
149         </P>
150         <OL>
151           <LI>Write the test case as a method in the fixture class</LI>
152           <LI>Create a TestCaller which runs that particular method</LI>
153         </OL>
154     
155     <P>Here is our test case class with a few extra case methods:</P>
156     <TT><PRE>&#9;class ComplexNumberTest : public TestCase  {
157     &#9;private:
158             Complex &#9;*m_10_1, *m_1_1; *m_11_2;
159     &#9;public:
160     &#9;void&#9;&#9;setUp ()  {
161     &#9;&#9;&#9;    m_10_1 = new Complex (10, 1);
162     &#9;&#9;&#9;    m_1_1  = new Complex (1, 1);
163     &#9;&#9;&#9;    m_11_2 = new Complex (11, 2);  
164                             }
165     &#9;void&#9;&#9;tearDown ()  {
166     &#9;&#9;&#9;    delete m_10_1, delete m_1_1, delete m_11_2;
167     &#9;&#9;&#9;}
168     &#9;void&#9;&#9;testEquality ()  {
169     &#9;&#9;&#9;    CPPUNIT_ASSERT (*m_10_1 == *m_10_1);
170     &#9;&#9;&#9;    CPPUNIT_ASSERT (!(*m_10_1 == *m_11_2));
171     &#9;&#9;&#9;}
172     &#9;void&#9;&#9;testAddition ()  {
173     &#9;&#9;&#9;    CPPUNIT_ASSERT (*m_10_1 + *m_1_1 == *m_11_2);
174                      &#9;}
175     &#9;};</PRE>
176     </TT><P>One may create and run instances for each test case like this:</P>
177     <TT><PRE>&#9;TestCaller&lt;ComplexNumberTest&gt; test("testEquality", &amp;ComplexNumberTest::testEquality);
178             test.run (); </PRE>
179     </TT><P>The second argument to the test caller constructor is the address of a method on ComplexNumberTest. When the test caller is run, that specific method will be run.  This is not a useful thing to do, however, as no diagnostics will be
180         displayed.  One will normally use a TestRunner (see below) to display
181         the results.
182         </P>
183     
184         <P>Once you have several tests, organize them into a suite.</P>
185     <P>&nbsp;</P>
186     <H2>Suite</H2>
187     <P>How do you set up your tests so that you can run them all at once?<BR>
188     <BR>
189     CppUnit provides a TestSuite class that runs any number of TestCases together.
190         We saw, above, how to run a single test case.
191     To create a suite of two or more tests, you do the following:</P>
192     <TT><PRE>&#9;TestSuite suite;
193     &#9;TestResult result;
194     &#9;suite.addTest (new TestCaller&lt;ComplexNumberTest&gt;("testEquality", &amp;ComplexNumberTest::testEquality));
195     &#9;suite.addTest (new TestCaller&lt;ComplexNumberTest&gt;("testAddition", &amp;ComplexNumberTest::testAddition));
196     &#9;suite.run (&amp;result);
197                </PRE>
198     </TT><P>TestSuites don't only have to contain callers for TestCases. They can contain any object that implements the Test interface. For example, you can create a TestSuite in your code and I can create one in mine, and we can run them together by creating a TestSuite that contains both: </P>
199     <TT><PRE>&#9;TestSuite suite;
200     &#9;TestResult result;
201     &#9;suite.addTest (ComplexNumberTest::suite ());
202     &#9;suite.addTest (SurrealNumberTest::suite ());
203     &#9;suite.run (&amp;result);</PRE>
204     </TT><P>&nbsp;</P>
205     <H2>TestRunner</H2>
206     <P><b>FIXME:</b> check over; MSVC6 only.
207     How do you run your tests and collect their results? </P>
208     <P>Once you have a test suite, you'll want to run it. CppUnit provides tools to define the suite to be run and to display its results. You make your suite accessible to a TestRunner program with a static method <I>suite</I> that returns a test suite. <BR>
209     For example, to make a ComplexNumberTest suite available to a TestRunner, add the following code to ComplexNumberTest: </P>
210     <TT><PRE>&#9;public: static Test *suite ()  {
211     &#9;    TestSuite *suiteOfTests = new TestSuite;
212     &#9;    suiteOfTests-&gt;addTest (new TestCaller&lt;ComplexNumberTest&gt;("testEquality", testEquality));
213     &#9;    suiteOfTests-&gt;addTest (new TestCaller&lt;ComplexNumberTest&gt;("testAddition", testAddition));
214                 return suiteOfTests;
215     &#9;}</PRE>
216     </TT>
217     <P>To use the text version, include the header file for the test in TestRunner.cpp:</P>
218     <TT><PRE>&#9;#include "ExampleTestCase.h"
219     &#9;#include "ComplexNumberTest.h"</PRE>
220     </TT><P>And add a call to "addTest (string, Test *)" in the "main ()" function:</P>
221     <TT><PRE>&#9;int main (int ac, char **av)  {
222     &#9;    TestRunner runner;
223     &#9;    runner.addTest (ExampleTestCase::suite ());
224     &#9;    runner.addTest (ComplexNumberTest::suite ());
225     &#9;    runner.run ();
226     &#9;    return 0;
227     &#9;}</PRE>
228     </TT><P>The TestRunner will run the tests. If all the tests pass, you'll get an informative message. If any fail, you'll get the following information:</P>
229     <OL>
230     
231     <LI>The name of the test case that failed</LI>
232     <LI>The name of the source file that contains the test</LI>
233     <LI>The line number where the failure occurred</LI>
234     <LI>All of the text inside the call to CPPUNIT_ASSERT which detected the failure</LI></OL>
235     
236     <P>CppUnit distinguishes between <I>failures</I> and <I>errors</I>. A failure is anticipated and checked for with assertions. Errors are unanticipated problems like division by zero and other exceptions thrown by the C++ runtime or your code.</P>
